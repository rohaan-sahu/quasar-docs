title: Events
description: Emitting on-chain events via log-based and self-CPI mechanisms.

Quasar supports two event emission strategies with different cost and security tradeoffs:

- **`emit!()`** -- emits via `sol_log_data` at ~100 CU. Fast but spoofable: any program can emit logs that look like another program's events.
- **`emit_cpi!()`** -- emits via self-CPI at ~1,000 CU. The program's own ID appears in the transaction trace, proving the event originated from the program.

Both paths use the same event struct definition. You choose the emission method at the call site.

## Defining events

Use the `#[event]` attribute macro on a struct with named fields. Each event requires an explicit discriminator:

```rust
use quasar_lang::prelude::*;

#[event(discriminator = 0)]
pub struct MakeEvent {
    pub escrow: Address,
    pub maker: Address,
    pub mint_a: Address,
    pub mint_b: Address,
    pub deposit: u64,
    pub receive: u64,
}

#[event(discriminator = 1)]
pub struct TakeEvent {
    pub escrow: Address,
}

#[event(discriminator = 2)]
pub struct RefundEvent {
    pub escrow: Address,
}
```

This is the event file from the escrow example. Each event gets a unique discriminator within the program, used to identify the event type when parsing transaction logs.

### Supported field types

Event structs support a fixed set of field types:

| Type | Size |
|------|------|
| `u8`, `i8`, `bool` | 1 byte |
| `u16`, `i16` | 2 bytes |
| `u32`, `i32` | 4 bytes |
| `u64`, `i64` | 8 bytes |
| `u128`, `i128` | 16 bytes |
| `Address` | 32 bytes |

No other types are allowed. Strings, vectors, and custom structs are not supported in events. The macro rejects unsupported types at compile time with a clear error message.

### What the macro generates

The `#[event]` macro generates:

1. A `#[repr(C)]` attribute on the struct, ensuring a deterministic field layout.
2. A compile-time assertion that the struct has **no padding**: `assert!(size_of::<Event>() == computed_data_size)`. This is critical because the serialization uses `memcpy` (via `copy_nonoverlapping`) -- padding bytes would leak uninitialized memory.
3. An implementation of the `Event` trait with `write_data` (memcpy serialization) and `emit` (self-CPI format).
4. An `emit_log` method for log-based emission.

The `Event` trait is defined in `quasar_lang::traits`:

```rust
pub trait Event {
    const DISCRIMINATOR: &'static [u8];
    const DATA_SIZE: usize;
    fn write_data(&self, buf: &mut [u8]);
    fn emit(&self, f: impl FnOnce(&[u8]) -> Result<(), ProgramError>) -> Result<(), ProgramError>;
}
```

## Log-based emission with `emit!()`

The `emit!()` macro calls `sol_log_data` to write the event data to the transaction log. This is the cheapest emission path at approximately 100 CU:

```rust
impl<'info> Make<'info> {
    #[inline(always)]
    pub fn emit_event(&self, deposit: u64, receive: u64) -> Result<(), ProgramError> {
        emit!(MakeEvent {
            escrow: *self.escrow.address(),
            maker: *self.maker.address(),
            mint_a: *self.mint_a.address(),
            mint_b: *self.mint_b.address(),
            deposit,
            receive,
        });
        Ok(())
    }
}
```

The `emit!()` macro expands to `$event.emit_log()`. The generated `emit_log` method:

1. Allocates a stack buffer of `discriminator_len + data_size` bytes via `MaybeUninit`.
2. Copies the discriminator bytes to the start of the buffer.
3. Calls `write_data` to memcpy the struct fields after the discriminator.
4. Passes the buffer to `sol_log_data` (the `log_data` function in `quasar_lang::log`).

The wire format for log-based events is: `[discriminator bytes] [struct data]`.

### When to use `emit!()`

Use `emit!()` when:

- CU budget is tight and you want the cheapest possible emission (~100 CU).
- The event consumer trusts the transaction's program invocation chain (e.g., your own indexer watching a known program).
- Spoofing resistance is not required.

## Self-CPI emission with `emit_cpi!()`

The `emit_cpi!()` macro emits events via a self-CPI -- the program invokes itself with the event data as instruction data. Because the program's ID appears in the transaction's instruction trace, indexers can verify the event was genuinely emitted by the program.

### The Event Authority PDA

Self-CPI events require an **Event Authority** -- a PDA derived from the seed `"__event_authority"`. The `#[program]` macro automatically generates this PDA type and its bump seed:

```rust
// Generated by #[program]
pub struct EventAuthority;

impl EventAuthority {
    pub const BUMP: u8 = /* computed at compile time or init */;
}
```

To use `emit_cpi!()`, your instruction's `Accounts` struct must include both the program account and the event authority:

```rust
#[derive(Accounts)]
pub struct EmitViaCpi<'info> {
    pub signer: &'info Signer,
    pub event_authority: &'info EventAuthority,
    pub program: &'info Program<QuasarTestEvents>,
}
```

### Using `emit_cpi!()`

The macro must be called inside an instruction handler with access to `self.program` and `self.event_authority`:

```rust
impl<'info> EmitViaCpi<'info> {
    #[inline(always)]
    pub fn handler(&self, value: u64) -> Result<(), ProgramError> {
        emit_cpi!(SimpleEvent { value })?;
        Ok(())
    }
}
```

The `emit_cpi!()` macro expands to:

```rust
self.program.emit_event(&event, self.event_authority, crate::EventAuthority::BUMP)
```

This calls `emit_event_cpi` internally, which:

1. Constructs the event data with a `0xFF` prefix byte followed by the discriminator and struct data.
2. Creates a single CPI account (the event authority as a readonly signer).
3. Signs the CPI with the event authority PDA seeds: `["__event_authority", &[bump]]`.
4. Invokes the program itself via `sol_invoke_signed_c`.

The wire format for self-CPI events is: `[0xFF] [discriminator bytes] [struct data]`.

The `0xFF` prefix byte is reserved -- the `#[program]` macro generates a compile-time assertion that no instruction discriminator equals `0xFF`, ensuring the self-CPI event data cannot collide with regular instruction dispatch.

### When to use `emit_cpi!()`

Use `emit_cpi!()` when:

- You need **spoofing resistance**: indexers must be able to verify the event was emitted by your program, not by a malicious program that forged log data.
- The additional ~900 CU cost (approximately 1,000 CU total) is acceptable.
- The instruction already has the event authority and program accounts available.

### Cost comparison

| Method | Approximate CU | Spoofing resistant | Accounts required |
|--------|---------------|-------------------|-------------------|
| `emit!()` | ~100 CU | No | None |
| `emit_cpi!()` | ~1,000 CU | Yes | `EventAuthority`, `Program<Self>` |

## Serialization details

Event serialization uses `copy_nonoverlapping` (memcpy) from the struct directly to the output buffer. This works because:

1. The struct is `#[repr(C)]`, so field layout is deterministic.
2. The no-padding assertion guarantees `size_of::<Event>() == sum of field sizes`.
3. All field types are fixed-size with alignment 1 (or have alignment 1 in their in-memory representation).

This approach is significantly cheaper than Borsh serialization -- a single memcpy versus per-field serialization calls.

```rust
// Generated write_data implementation
fn write_data(&self, buf: &mut [u8]) {
    unsafe {
        core::ptr::copy_nonoverlapping(
            self as *const Self as *const u8,
            buf.as_mut_ptr(),
            DATA_SIZE,
        );
    }
}
```

## Discriminator management

Event discriminators are developer-specified integers, just like account and instruction discriminators. Within a program, each event must have a unique discriminator:

```rust
#[event(discriminator = 0)]
pub struct MakeEvent { ... }

#[event(discriminator = 1)]
pub struct TakeEvent { ... }

#[event(discriminator = 2)]
pub struct RefundEvent { ... }
```

Discriminators can be single bytes or multi-byte arrays. Single-byte discriminators are the common case and the most efficient.

Event discriminators occupy a separate namespace from instruction and account discriminators -- an event with `discriminator = 0` does not conflict with an instruction with `discriminator = 0`.

## Empty events

Events with no fields are supported. They emit only the discriminator:

```rust
#[event(discriminator = 5)]
pub struct EmptyEvent {}
```

This can be useful as a signal that an action occurred, without any associated data.

## Parsing events off-chain

When parsing transaction logs, events emitted via `emit!()` appear as base64-encoded `sol_log_data` entries. The format is:

```
[discriminator: N bytes] [data: DATA_SIZE bytes]
```

Events emitted via `emit_cpi!()` appear as instruction data in the transaction's inner instructions. The format includes the `0xFF` prefix:

```
[0xFF] [discriminator: N bytes] [data: DATA_SIZE bytes]
```

To decode an event:

1. Read the discriminator bytes to identify the event type.
2. Cast the remaining bytes to the event's `#[repr(C)]` struct (interpreting multi-byte integers as little-endian).

The Quasar IDL includes event definitions with field types and discriminators, enabling client libraries to generate event parsers automatically.
