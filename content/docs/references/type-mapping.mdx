title: Type Mapping
description: Mapping between Rust types and their zero-copy Pod representations.
---

Quasar uses zero-copy account access, which means account data is read directly from the runtime's memory buffer without deserialization. To make this safe and portable, native Rust integer types are mapped to alignment-1 Pod types that store values as little-endian byte arrays.

## Rust Type to Pod Type Mapping

When you define an `#[account]` struct, the derive macro generates a companion `#[repr(C)]` struct where each field is mapped to its zero-copy equivalent:

| Rust Type | Zero-Copy Type | Size (bytes) | Alignment |
|-----------|---------------|--------------|-----------|
| `u8` | `u8` (no wrapper) | 1 | 1 |
| `i8` | `i8` (no wrapper) | 1 | 1 |
| `u16` | `PodU16` | 2 | 1 |
| `i16` | `PodI16` | 2 | 1 |
| `u32` | `PodU32` | 4 | 1 |
| `i32` | `PodI32` | 4 | 1 |
| `u64` | `PodU64` | 8 | 1 |
| `i64` | `PodI64` | 8 | 1 |
| `u128` | `PodU128` | 16 | 1 |
| `i128` | `PodI128` | 16 | 1 |
| `bool` | `PodBool` | 1 | 1 |
| `Address` | `Address` (no wrapper) | 32 | 1 |

All Pod types have alignment 1, which is critical for zero-copy access. The Solana runtime provides account data in a flat byte buffer with no alignment guarantees, so all types used in `#[repr(C)]` account structs must have alignment 1 to avoid undefined behavior from misaligned pointer casts.

### Example

Given this account definition:

```rust
#[account(discriminator = 1)]
pub struct Vault {
    pub authority: Address,
    pub balance: u64,
    pub is_active: bool,
}
```

The derive macro generates this zero-copy companion:

```rust
#[repr(C)]
#[derive(Copy, Clone)]
pub struct VaultZc {
    pub authority: Address,     // 32 bytes
    pub balance: PodU64,        // 8 bytes
    pub is_active: PodBool,     // 1 byte
}
```

The companion struct has a compile-time assertion that its alignment is 1:

```rust
const _: () = assert!(core::mem::align_of::<VaultZc>() == 1);
```

## Pod Types

Pod types are defined in the `quasar_pod` crate and re-exported via `quasar_lang::prelude`. Each is `#[repr(transparent)]` over `[u8; N]`, storing values in little-endian byte order.

### Reading Values

Use `.get()` to extract the native Rust value:

```rust
let balance: u64 = vault.balance.get();
let is_active: bool = vault.is_active.get();
let slot: u64 = clock.slot.get();
```

### Writing Values

When writing to a mutable account, you can assign native values directly. The `#[account]` macro generates setters that handle the conversion:

```rust
// Direct field access via DerefMut
vault.balance = PodU64::from(1000u64);
vault.is_active = PodBool::from(true);
```

### Arithmetic

Pod types implement standard arithmetic operators (`+`, `-`, `*`, `/`, `%`) that work with both Pod and native types:

```rust
let a = PodU64::from(100u64);
let b = a + 50u64;           // PodU64 + u64 -> PodU64
let c = a + PodU64::from(50); // PodU64 + PodU64 -> PodU64
let d = a * 2u64;            // PodU64 * u64 -> PodU64
```

**Overflow behavior:**
- **Debug builds:** Panics on overflow (via `checked_*` operations)
- **Release builds:** Uses wrapping semantics for CU efficiency

For explicit overflow detection in all builds, use the checked methods:

| Method | Description |
|--------|-------------|
| `checked_add(rhs)` | Returns `None` on overflow |
| `checked_sub(rhs)` | Returns `None` on underflow |
| `checked_mul(rhs)` | Returns `None` on overflow |
| `checked_div(rhs)` | Returns `None` if `rhs` is zero |
| `saturating_add(rhs)` | Clamps at numeric bounds |
| `saturating_sub(rhs)` | Clamps at zero (unsigned) or bounds (signed) |
| `saturating_mul(rhs)` | Clamps at numeric bounds |

```rust
let balance = vault.balance;
let new_balance = balance.checked_add(deposit_amount)
    .ok_or(ProgramError::ArithmeticOverflow)?;
```

### Constants

Each Pod type provides:

| Constant | Description |
|----------|-------------|
| `ZERO` | The zero value |
| `MAX` | The largest representable value |
| `MIN` | The smallest representable value |

```rust
if vault.balance == PodU64::ZERO {
    return Err(MyError::EmptyVault.into());
}
```

### Comparisons

Pod types implement `PartialEq`, `Eq`, `PartialOrd`, and `Ord`. You can compare a Pod value directly with its native type:

```rust
if vault.balance > 100u64 {
    // ...
}
if vault.balance == PodU64::ZERO {
    // ...
}
```

### PodBool

`PodBool` stores a boolean as a single byte. Any non-zero byte is considered `true` (canonical form is `0x01`).

```rust
let active: bool = vault.is_active.get();  // read
vault.is_active = PodBool::from(true);     // write
let toggled = !vault.is_active;            // bitwise NOT
```

### Bitwise Operations

Pod integer types support bitwise operators:

| Operator | Description |
|----------|-------------|
| `&` | Bitwise AND |
| `\|` | Bitwise OR |
| `^` | Bitwise XOR |
| `!` | Bitwise NOT |
| `<<` | Left shift |
| `>>` | Right shift |

## Discriminator

Every account and instruction type has a discriminator prefix -- a byte sequence written at the start of the account data that identifies the account type.

### Account Discriminators

Set via the `#[account]` macro:

```rust
#[account(discriminator = 1)]
pub struct Vault { ... }

#[account(discriminator = [1, 2])]
pub struct Escrow { ... }
```

Rules:
- At least one byte is required
- At least one byte must be non-zero (all-zero discriminators are rejected at compile time, since they are indistinguishable from uninitialized account data)
- Single values are stored as a one-byte discriminator
- Arrays support multi-byte discriminators

### Instruction Discriminators

Set via the `#[instruction]` macro:

```rust
#[instruction(discriminator = 0)]
pub fn initialize(ctx: Ctx<Initialize>) -> Result<(), ProgramError> { ... }

#[instruction(discriminator = 1)]
pub fn deposit(ctx: Ctx<Deposit>) -> Result<(), ProgramError> { ... }
```

Instruction discriminators can be zero. The `quasar new instruction` command auto-increments from the highest existing discriminator.

## Calculating Account Space

Total account space is calculated as:

```
space = discriminator_size + fixed_fields_size + dynamic_prefix_bytes + dynamic_data_bytes
```

### Fixed-Size Accounts

For accounts with only fixed-width fields, the `Space` trait provides a compile-time constant:

```rust
#[account(discriminator = 1)]
pub struct Vault {
    pub authority: Address,  // 32 bytes
    pub balance: u64,        // 8 bytes
    pub is_active: bool,     // 1 byte
}
// Vault::SPACE = 1 (disc) + 32 + 8 + 1 = 42 bytes
```

The formula:

```
SPACE = discriminator_bytes + sizeof(ZcCompanion)
```

Where `sizeof(ZcCompanion)` is the sum of all field sizes after Pod mapping:

| Field | Pod Type | Size |
|-------|----------|------|
| `authority: Address` | `Address` | 32 |
| `balance: u64` | `PodU64` | 8 |
| `is_active: bool` | `PodBool` | 1 |
| **Total fields** | | **41** |
| **+ discriminator (1 byte)** | | **42** |

### Dynamic Accounts

For accounts with `String` or `Vec` fields, space is variable. The macro provides `MIN_SPACE` and `MAX_SPACE` constants:

```rust
#[account(discriminator = 5)]
pub struct Profile<'a> {
    pub owner: Address,              // 32 bytes (fixed)
    pub name: String<u16, 64>,       // 2 prefix + 0..64 data
    pub tags: Vec<Address, u8, 10>,  // 1 prefix + 0..320 data
}
// Profile::MIN_SPACE = 1 (disc) + 32 (fixed) + 2 + 1 = 36 bytes
// Profile::MAX_SPACE = 36 + 64 + 320 = 420 bytes
```

Dynamic field storage:

| Field Type | Prefix | Data Bytes |
|------------|--------|------------|
| `String` | `u32` (4 bytes) | 0..1024 |
| `String<u16, N>` | `u16` (2 bytes) | 0..N |
| `String<u8, N>` | `u8` (1 byte) | 0..N |
| `Vec<T>` | `u32` (4 bytes) | 0..8 * sizeof(T) |
| `Vec<T, u16, N>` | `u16` (2 bytes) | 0..N * sizeof(T) |
| `Vec<T, u8, N>` | `u8` (1 byte) | 0..N * sizeof(T) |
| `&str` (tail) | none | remaining bytes |
| `&[u8]` (tail) | none | remaining bytes |

The `String` and `Vec` types are marker types that configure the wire format:

```rust
// String<P, MAX> where P = prefix type, MAX = max byte length
pub name: String,             // String<u32, 1024> (defaults)
pub tag: String<u8, 32>,     // u8 prefix, max 32 bytes
pub bio: String<u16, 4096>,  // u16 prefix, max 4096 bytes

// Vec<T, P, MAX> where T = element, P = prefix type, MAX = max count
pub members: Vec<Address>,           // Vec<Address, u32, 8> (defaults)
pub scores: Vec<PodU64, u8, 4>,     // u8 prefix, max 4 elements
```

### Tail Fields

A tail field (`&str` or `&[u8]`) consumes all remaining bytes in the account data with no length prefix. It must be the last dynamic field. Maximum size is 1024 bytes.

```rust
#[account(discriminator = 5)]
pub struct Note<'a> {
    pub author: Address,
    pub content: &'a str,  // tail field -- no prefix, uses remaining data
}
```

### Space Override

Use `space = <expr>` on `init` to override the computed space:

```rust
#[account(init, payer = payer, space = 8 + 32 + 4 + name.len())]
pub profile: &'info mut Account<Profile<'info>>,
```
