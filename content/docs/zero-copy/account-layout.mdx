---
title: Account Layout
description: How Quasar lays out account data in memory for zero-copy access.
---

`#[account]` generates a **zero-copy companion struct** -- a `#[repr(C)]` struct where every field with alignment greater than 1 is replaced with its Pod equivalent. This companion struct overlays the raw account data buffer. Field access goes through `Deref`/`DerefMut`, so the translation is invisible at the call site.

## From source to companion struct

Given this `Escrow` account:

```rust
#[account(discriminator = 1)]
pub struct Escrow {
    pub maker: Address,
    pub mint_a: Address,
    pub mint_b: Address,
    pub maker_ta_b: Address,
    pub receive: u64,
    pub bump: u8,
}
```

The macro generates (simplified):

```rust
// The view type -- what you actually interact with
#[repr(transparent)]
pub struct Escrow {
    __view: AccountView,
}

// The ZC companion -- overlays the raw bytes
#[repr(C)]
#[derive(Copy, Clone)]
pub struct EscrowZc {
    pub maker: Address,         // [u8; 32] -- alignment 1
    pub mint_a: Address,        // [u8; 32] -- alignment 1
    pub mint_b: Address,        // [u8; 32] -- alignment 1
    pub maker_ta_b: Address,    // [u8; 32] -- alignment 1
    pub receive: PodU64,        // [u8; 8]  -- alignment 1
    pub bump: u8,               // u8       -- alignment 1
}

const _: () = assert!(core::mem::align_of::<EscrowZc>() == 1);
```

The companion struct is named `{Name}Zc`. You access fields through `Deref` on the view type.

## Type mapping table

| Rust type | ZC companion type | Size (bytes) | Alignment |
|-----------|-------------------|-------------|-----------|
| `u8` | `u8` | 1 | 1 |
| `i8` | `i8` | 1 | 1 |
| `u16` | `PodU16` | 2 | 1 |
| `i16` | `PodI16` | 2 | 1 |
| `u32` | `PodU32` | 4 | 1 |
| `i32` | `PodI32` | 4 | 1 |
| `u64` | `PodU64` | 8 | 1 |
| `i64` | `PodI64` | 8 | 1 |
| `u128` | `PodU128` | 16 | 1 |
| `i128` | `PodI128` | 16 | 1 |
| `bool` | `PodBool` | 1 | 1 |
| `Address` | `Address` | 32 | 1 |

Types that already have alignment 1 (`u8`, `i8`, `Address`) pass through unchanged. Types with higher native alignment (`u16` through `u128`, `bool`) are replaced with their Pod equivalents.

Any type not in this table (e.g., a custom `#[repr(C)]` struct) passes through unchanged. Ensure such types have alignment 1 -- the compile-time assertion on the companion struct will catch violations.

## Memory layout

Account data follows this layout:

```
[discriminator] [fixed fields (ZC companion)] [dynamic fields, if any]
```

For the `Escrow` account with `discriminator = 1`, the layout is:

```
Offset  Size   Field
------  ----   -----
0       1      discriminator (0x01)
1       32     maker: Address
33      32     mint_a: Address
65      32     mint_b: Address
97      32     maker_ta_b: Address
129     8      receive: PodU64
137     1      bump: u8
------  ----
Total: 138 bytes
```

The `Space` trait reports this total: `Space::SPACE = 1 + size_of::<EscrowZc>() = 1 + 137 = 138`.

Because the companion struct is `#[repr(C)]` and every field has alignment 1, there is **no padding** between fields. The layout is fully deterministic and matches the wire format byte-for-byte.

## AccountCheck: validation before pointer cast

`AccountCheck::check()` validates the account before any pointer cast:

```rust
impl AccountCheck for Escrow {
    fn check(view: &AccountView) -> Result<(), ProgramError> {
        let data = unsafe { view.borrow_unchecked() };
        // 1. Bounds check
        if data.len() < 1 + core::mem::size_of::<EscrowZc>() {
            return Err(ProgramError::AccountDataTooSmall);
        }
        // 2. Discriminator check
        if unsafe { *data.get_unchecked(0) } != 1 {
            return Err(ProgramError::InvalidAccountData);
        }
        Ok(())
    }
}
```

The bounds check ensures the buffer is large enough. The discriminator check verifies the bytes match. All-zero discriminators are banned at compile time (see [Safety Model](/docs/zero-copy/safety-model)), so uninitialized account data never passes validation.

## Multi-byte discriminators

Discriminators can be more than one byte. Single-byte discriminators are the common case.

```rust
#[account(discriminator = [1, 0, 0, 0])]
pub struct LargeAccount { ... }
```

## Generated trait implementations

`#[account]` generates implementations for:

### `Discriminator`

```rust
impl Discriminator for Escrow {
    const DISCRIMINATOR: &'static [u8] = &[1];
}
```

### `Space`

```rust
impl Space for Escrow {
    const SPACE: usize = 1 + core::mem::size_of::<EscrowZc>();
}
```

### `Owner`

```rust
impl Owner for Escrow {
    const OWNER: Address = crate::ID;
}
```

The owner is always the program that defines the account, taken from `declare_id!`.

### `set_inner`

Writes all field values into the account data buffer at once:

```rust
impl Escrow {
    pub fn set_inner(
        &mut self,
        maker: Address,
        mint_a: Address,
        mint_b: Address,
        maker_ta_b: Address,
        receive: u64,
        bump: u8,
    ) {
        let zc = /* pointer to ZC companion in account data */;
        zc.maker = maker;
        zc.mint_a = mint_a;
        zc.mint_b = mint_b;
        zc.maker_ta_b = maker_ta_b;
        zc.receive = PodU64::from(receive);
        zc.bump = bump;
    }
}
```

Native types like `u64` are automatically converted to their Pod equivalents when assigned. Address and `u8` pass through directly.

## Reading and writing fields

Field reads go through the zero-copy view:

```rust
// In an instruction handler
let maker: &Address = &escrow.maker;
let amount: PodU64 = escrow.receive;  // PodU64, not u64
let amount_native: u64 = escrow.receive.get();  // Convert to u64
```

Writing through `DerefMut`:

```rust
escrow.receive = PodU64::from(new_amount);
```

Writes go directly to the account data buffer -- no intermediate copy, no serialization step.

## Alignment-1 invariant

Every companion struct must have alignment 1. This is enforced at compile time:

```rust
const _: () = assert!(
    core::mem::align_of::<EscrowZc>() == 1,
    "ZC companion struct must have alignment 1; all fields must use Pod types or alignment-1 types"
);
```

If any field has alignment greater than 1, this assertion fires at compile time. See [Safety Model](/docs/zero-copy/safety-model) for the full picture.
