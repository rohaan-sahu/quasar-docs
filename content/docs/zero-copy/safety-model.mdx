description: How Quasar uses unsafe Rust safely -- pointer casts, interior mutability, compile-time assertions, and Miri validation.

Quasar uses `unsafe` for zero-copy account access, raw CPI syscalls, pointer arithmetic over the SVM input buffer, and `MaybeUninit` patterns for stack-allocated arrays. This page documents each pattern, its soundness argument, and its protections.

## Unsafe inventory

Five categories of `unsafe` in `quasar-lang`:

1. **Zero-copy pointer casts** (`&[u8]` to `&ZcType`)
2. **Interior mutability** (`&` to `&mut` casts)
3. **Raw CPI syscalls** (`sol_invoke_signed_c`)
4. **Pointer arithmetic** (remaining accounts, entrypoint dispatch)
5. **`MaybeUninit` patterns** (stack-allocated arrays)

## Zero-copy pointer casts

Every `Deref` on an account view type casts a raw byte buffer to a `#[repr(C)]` companion struct:

```rust
impl core::ops::Deref for Escrow {
    type Target = EscrowZc;

    fn deref(&self) -> &Self::Target {
        unsafe {
            &*(self.__view.data_ptr().add(DISC_LEN) as *const EscrowZc)
        }
    }
}
```

Soundness relies on three invariants:

1. **`#[repr(transparent)]`**: Account view types are `#[repr(transparent)]` wrappers over `AccountView`, so the cast is a no-op at the memory level.

2. **Alignment 1**: ZC companion structs use Pod types (`#[repr(transparent)]` over `[u8; N]`). Compile-time assertions enforce `align_of::<ZcStruct>() == 1`.

3. **Bounds checked upstream**: `AccountCheck::check()` validates `data_len >= size_of::<ZcStruct>() + discriminator_len` before any pointer cast.

## Interior mutability

`from_account_view_unchecked_mut` produces an `&mut Account<T>` that can coexist with `&AccountView` references to the same `RuntimeAccount`. All mutations go through raw pointer methods on `AccountView`, not through Rust's mutable reference semantics. Every call site checks `is_writable()` before performing the cast.

### Tree Borrows dependency

This pattern depends on **Tree Borrows** aliasing semantics:

- Miri must be run with `-Zmiri-tree-borrows`. Under the default Stacked Borrows model, the casts are flagged as UB.
- The Rust compiler does not currently commit to either aliasing model.
- This is an accepted risk shared with Pinocchio and other Solana frameworks that use the same pattern for zero-copy mutable account access.

## Raw CPI syscalls

CPI calls use `RawCpiBuilder` structs on the stack, layout-compatible with `CpiAccount` and transmuted before passing to `sol_invoke_signed_c`. Compile-time assertions enforce size, alignment, and field offset compatibility with the upstream `CpiAccount` and `RuntimeAccount` types.

## Pointer arithmetic

The remaining accounts system and entrypoint dispatch walk the SVM input buffer using raw pointer arithmetic. Every pointer dereference is preceded by a boundary check against the end of the accounts region, computed once at entrypoint dispatch.

## MaybeUninit patterns

`MaybeUninit` is used for stack-allocated arrays in CPI account construction, system instruction buffers, the remaining accounts iterator cache, declared account parsing, sysvar buffers, and PDA derivation. In all cases, elements are fully written before being read, with bounds checks preventing access to uninitialized memory.

## Compile-time assertions

Quasar uses `const _: () = assert!(...)` to catch invariant violations at compile time:

| Assertion | Purpose |
|-----------|---------|
| `align_of::<ZcStruct>() == 1` | ZC companion structs have alignment 1 |
| CPI builder size/alignment | Matches `CpiAccount` layout |
| Sysvar size/alignment | Correct `Rent`, `Clock` struct geometry |
| Discriminator non-zero | Prevents uninitialized data matching |
| No padding in event structs | Ensures memcpy serialization is safe |
| Instruction args alignment 1 | Safe for pointer cast from `&[u8]` |
| Return type alignment 1 | Return data pointer cast is aligned |
| Discriminator != `0xFF` | Reserved for self-CPI event prefix |
| No discriminator collisions | Same-kind collisions are compile errors |

## Miri validation

All unsafe code paths are tested under Miri with Tree Borrows and symbolic alignment checking:

```bash
MIRIFLAGS="-Zmiri-tree-borrows -Zmiri-symbolic-alignment-check" \
  cargo +nightly miri test -p quasar-lang --test miri
```

The test suite covers aliasing casts, CPI construction, event serialization, remaining accounts pointer arithmetic, and dynamic field boundary conditions.

## Attack mitigations

### All-zero discriminator attack

An attacker creates an account with all-zero data (the default for new accounts). If a program used all-zero discriminators, uninitialized accounts would pass validation.

**Mitigation**: All-zero discriminators are banned at compile time. Since uninitialized account data is all-zero, it can never match a valid discriminator.

### Re-initialization attack

An attacker calls an init instruction on an account that already contains valid data, overwriting it.

**Mitigation**: `init()` checks that the discriminator region is all zeros before writing. Valid accounts always have non-zero discriminators, so `init()` only succeeds on uninitialized accounts.

### Account revival attack

After closing an account within a transaction, a subsequent instruction re-opens it. If the data was not zeroed during close, the old discriminator persists.

**Mitigation**: `Account::close()` zeros the discriminator bytes before draining lamports, then reassigns the owner to the system program and resizes data to zero. Even within the same transaction, the account cannot pass discriminator validation after close.

### Misaligned access

Account data from the SVM has alignment 1. Casting to a struct with alignment > 1 is undefined behavior.

**Mitigation**: All zero-copy types use Pod fields with alignment 1. Every ZC companion struct has a compile-time assertion: `assert!(align_of::<ZcStruct>() == 1)`.

### Duplicate account chain overflow

Malformed SVM input with chained duplicate references could cause excessive iteration.

**Mitigation**: `resolve_dup_walk` uses iterative resolution with a 2-hop depth limit.

## CPI without borrow checks

Quasar's CPI path does not check or modify `RuntimeAccount::borrow_state` before invoking `sol_invoke_signed_c`. This is deliberate:

1. **Raw pointer access only**: Quasar accesses data through raw pointers, never through `try_borrow()`. The `borrow_state` field stays at its initial `NOT_BORROWED` value.

2. **No-duplicate enforcement**: `parse_accounts` checks that byte 0 of each `RuntimeAccount` header is `0xFF` (`NOT_BORROWED`). Duplicates have a small index number in byte 0, so the check rejects them.

3. **Zero-copy eliminates sync gap**: Quasar writes directly to `RuntimeAccount` -- no serialize-before-CPI / deserialize-after-CPI needed.

## Zero heap allocation

The `no_alloc!()` macro installs a global allocator that panics on any heap allocation. Any accidental `Vec`, `String`, or `Box` usage triggers a panic during testing. Use `heap_alloc!()` instead if your program needs heap access.
