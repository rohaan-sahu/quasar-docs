The Token-2022 program (also called Token Extensions) is the next-generation SPL token program. It is a superset of the original SPL Token program, with the same base account layouts but support for extensions like transfer fees, confidential transfers, and metadata pointers. Quasar provides dedicated types for Token-2022 as well as a unified interface abstraction that accepts either program.

## Token-2022 types

The `quasar-spl` crate exports `Token2022` and `Mint2022` marker types that work identically to `Token` and `Mint`, but validate that the account owner is the Token-2022 program instead of SPL Token:

```rust
use quasar_spl::{Token2022, Mint2022, TokenCpi};

#[derive(Accounts)]
pub struct Transfer2022<'info> {
    pub authority: &'info Signer,
    pub from: &'info mut Account<Token2022>,
    pub to: &'info mut Account<Token2022>,
    pub token_program: &'info Program<Token2022>,
}

impl<'info> Transfer2022<'info> {
    pub fn handler(&self, amount: u64) -> Result<(), ProgramError> {
        self.token_program
            .transfer(self.from, self.to, self.authority, amount)
            .invoke()
    }
}
```

Under the hood, `Token2022` and `Mint2022` use the same `TokenAccountState` (165 bytes) and `MintAccountState` (82 bytes) zero-copy layouts. The base token and mint data occupies the same first N bytes for both programs -- extensions live after the base data and do not affect the prefix layout.

Both `Program<Token2022>` and `Program<Token>` implement the `TokenCpi` trait, so all the same CPI methods are available: `transfer()`, `transfer_checked()`, `mint_to()`, `burn()`, `approve()`, `revoke()`, `close_account()`, and `sync_native()`.

## InterfaceAccount

When your program needs to accept token accounts from **either** SPL Token or Token-2022, use `InterfaceAccount<T>` instead of `Account<T>`:

```rust
use quasar_spl::{InterfaceAccount, Token, TokenCpi, TokenInterface};

#[derive(Accounts)]
pub struct InterfaceTransfer<'info> {
    pub authority: &'info Signer,
    pub from: &'info mut InterfaceAccount<Token>,
    pub to: &'info mut InterfaceAccount<Token>,
    pub token_program: &'info Interface<TokenInterface>,
}

impl<'info> InterfaceTransfer<'info> {
    pub fn handler(&self, amount: u64) -> Result<(), ProgramError> {
        self.token_program
            .transfer(self.from, self.to, self.authority, amount)
            .invoke()
    }
}
```

`InterfaceAccount<Token>` performs the following validation during account parsing:

1. Checks that the account owner is either SPL Token (`TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA`) or Token-2022 (`TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb`).
2. Runs `T::check()` to validate the account data length (at least 165 bytes for token accounts, 82 bytes for mints).
3. For mutable references, additionally checks that the account is writable.

Like `Account<T>`, `InterfaceAccount<T>` implements `Deref` and `DerefMut` to the underlying zero-copy type, so you can access fields directly:

```rust
let balance = self.from.amount();
let mint_addr = self.from.mint();
```

The same interface pattern works for mints:

```rust
pub mint: &'info InterfaceAccount<Mint>,
```

## TokenInterface

The `Interface<TokenInterface>` type is the program account counterpart to `InterfaceAccount`. Where `Program<Token>` only accepts the SPL Token program, `Interface<TokenInterface>` accepts either:

```rust
pub token_program: &'info Interface<TokenInterface>,
```

At parse time, `TokenInterface` validates that the account address matches one of the two known token program addresses. The `TokenCpi` trait is implemented for `Interface<TokenInterface>`, so all CPI methods work the same way:

```rust
self.token_program
    .transfer(self.from, self.to, self.authority, amount)
    .invoke()?;
```

The SVM routes the CPI to whichever program was actually passed in the transaction. Your instruction code stays identical regardless of which token program the user provides.

## When to use which type

Choose the right combination based on your program's requirements:

| Scenario | Account type | Program type |
|----------|-------------|-------------|
| Only SPL Token support | `Account<Token>` / `Account<Mint>` | `Program<Token>` |
| Only Token-2022 support | `Account<Token2022>` / `Account<Mint2022>` | `Program<Token2022>` |
| Both programs supported | `InterfaceAccount<Token>` / `InterfaceAccount<Mint>` | `Interface<TokenInterface>` |

If you are building a new program and do not have a strong reason to restrict to one token program, prefer the `InterfaceAccount` + `TokenInterface` combination. It makes your program forward-compatible without any code changes.

## Token initialization with interfaces

The `token::mint` and `token::authority` attributes work with both direct types and interface types. The token program used for initialization is inferred from whichever program account is present in the accounts struct:

```rust
#[derive(Accounts)]
pub struct InitVault<'info> {
    pub payer: &'info mut Signer,
    #[account(init, token::mint = mint, token::authority = authority)]
    pub vault: &'info mut InterfaceAccount<Token>,
    pub mint: &'info InterfaceAccount<Mint>,
    pub authority: &'info Signer,
    pub token_program: &'info Interface<TokenInterface>,
    pub system_program: &'info Program<System>,
}
```

The same `InitToken` and `InitMint` traits are bounded on `TokenCpi`, which is implemented by all three program wrappers.

## TokenClose with interfaces

The `TokenClose` trait is implemented for all interface variants:

```rust
use quasar_spl::{InterfaceAccount, Token, TokenClose};

// Works with InterfaceAccount just like Account
self.vault.close(self.token_program, self.destination, self.authority)
    .invoke_signed(&seeds)?;
```

## Extension compatibility

The zero-copy types in Quasar read the base token/mint data (the first 165 or 82 bytes). Token-2022 extensions are stored **after** the base data and do not affect the fields accessible through `TokenAccountState` or `MintAccountState`.

If you need to read extension data, you can access the raw account data through the `AccountView`:

```rust
let view = self.token_account.to_account_view();
let raw_data = unsafe {
    core::slice::from_raw_parts(view.data_ptr(), view.data_len())
};
// Extension data starts at offset 165 for token accounts
// Parse extension-specific layouts as needed
```

For most use cases -- transfers, minting, burning, closing -- the base data and standard CPI methods are sufficient. The extensions are handled by the Token-2022 program itself during CPI execution.

## Migrating from Token to Interface types

If you have an existing program using `Account<Token>` and `Program<Token>`, upgrading to support both programs is a straightforward type substitution:

```rust
// Before: SPL Token only
pub from: &'info mut Account<Token>,
pub token_program: &'info Program<Token>,

// After: both programs
pub from: &'info mut InterfaceAccount<Token>,
pub token_program: &'info Interface<TokenInterface>,
```

No changes to your business logic are needed. The CPI methods have identical signatures across all three program wrappers.
