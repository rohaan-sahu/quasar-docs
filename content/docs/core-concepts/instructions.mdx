The `#[instruction]` macro generates discriminator matching, argument parsing, and context construction for each handler.

## Defining an Instruction

Each instruction is a function annotated with `#[instruction(discriminator = N)]` inside a `#[program]` module:

```rust
#[program]
mod quasar_vault {
    use super::*;

    #[instruction(discriminator = 0)]
    pub fn deposit(ctx: Ctx<Deposit>, amount: u64) -> Result<(), ProgramError> {
        ctx.accounts.deposit(amount)
    }

    #[instruction(discriminator = 1)]
    pub fn withdraw(ctx: Ctx<Withdraw>, amount: u64) -> Result<(), ProgramError> {
        ctx.accounts.withdraw(amount)
    }
}
```

Each instruction function must:

1. Take `ctx: Ctx<T>` or `ctx: CtxWithRemaining<T>` as its first parameter.
2. Specify a `discriminator` value in the attribute.
3. Return `Result<(), ProgramError>` (or `Result<T, ProgramError>` for return data).

## Discriminators

Discriminators are developer-specified integers with a predictable wire format.

```rust
#[instruction(discriminator = 0)]    // instruction data starts with [0x00]
pub fn deposit(...) -> Result<(), ProgramError> { ... }

#[instruction(discriminator = 1)]    // instruction data starts with [0x01]
pub fn withdraw(...) -> Result<(), ProgramError> { ... }
```

### Discriminator Rules

- **Same length**: All discriminators in a program must have the same byte length. Single-byte gives you 255 possible instructions (0x00 through 0xFE).
- **No duplicates**: Checked at compile time.
- **No 0xFF prefix**: Reserved for the event self-CPI protocol. Compile error if violated.
- **Multi-byte**: For programs needing more than 255 instructions: `discriminator = [0, 1]`.

### Dispatch

The discriminator prefix is matched against the handler table. `0xFF` routes to the event handler. Unmatched discriminators return `ProgramError::InvalidInstructionData`.

## Arguments

After the discriminator prefix, the remaining instruction data contains the function arguments.

### Fixed-Size Arguments

Fixed-size arguments (`u64`, `Address`, `bool`, etc.) are read directly from the instruction data buffer with no copying.

```rust
#[instruction(discriminator = 0)]
pub fn deposit(ctx: Ctx<Deposit>, amount: u64) -> Result<(), ProgramError> {
    ctx.accounts.deposit(amount)
}
```

### Dynamic Arguments

Dynamic arguments use length-prefixed encoding with `String<P, MAX>` and `Vec<T, P, MAX>` (where `P` is `u8`, `u16`, or `u32`).

```rust
#[instruction(discriminator = 2)]
pub fn set_label(ctx: Ctx<SetLabel>, label: String<32>) -> Result<(), ProgramError> {
    ctx.accounts.update_label(label)
}
```

`String<32>` uses the default `u32` prefix. The generated code reads the prefix, validates length and UTF-8, and produces a `&str` into the instruction data buffer. `Vec<T, P, MAX>` works the same way, producing a `&[T]` slice. Element types must have alignment 1 (enforced at compile time).

### Tail Arguments

A tail argument consumes all remaining instruction data after fixed fields. Use `&[u8]` or `&str`:

```rust
#[instruction(discriminator = 0)]
pub fn process(ctx: Ctx<Process>, data: &[u8]) -> Result<(), ProgramError> {
    // data contains all remaining bytes
}
```

Only one tail argument is allowed, and it must be the last parameter. `&str` tails additionally validate UTF-8.

### Wire Format Summary

The instruction data layout for `deposit(ctx, amount: u64)` with `discriminator = 0`:

```text
[0x00]                    <- discriminator (1 byte)
[amount: 8 bytes LE]      <- u64 argument
```

For `set_label(ctx, label: String<32>)` with `discriminator = 2` (default `u32` prefix):

```text
[0x02]                         <- discriminator (1 byte)
[len: 4 bytes LE]              <- u32 string length prefix
[label: len bytes UTF-8]       <- string data
```

## Context Types

### `Ctx<'info, T>`

```rust
pub struct Ctx<'info, T: ParseAccounts<'info> + AccountCount> {
    pub accounts: T,
    pub bumps: T::Bumps,
    pub program_id: &'info [u8; 32],
    pub data: &'info [u8],        // instruction data, discriminator already consumed
}
```

### `CtxWithRemaining<'info, T>`

Same fields as `Ctx`, plus a `remaining_accounts()` method that returns a `RemainingAccounts` iterator over accounts past the declared set:

```rust
pub fn create(ctx: CtxWithRemaining<Create>, threshold: u8) -> Result<(), ProgramError> {
    ctx.accounts.create_multisig(threshold, &ctx.bumps, ctx.remaining_accounts())
}
```

## Return Data

Return data to the caller by using `Result<T, ProgramError>` where `T` is a fixed-size, alignment-1 type:

```rust
#[instruction(discriminator = 0)]
pub fn compute_value(ctx: Ctx<Compute>) -> Result<PodU64, ProgramError> {
    Ok(PodU64::from(42u64))
}
```

Close operations run before return data is set. Alignment 1 on the return type is enforced at compile time.

## Instruction Data Access in Account Validation

Use `#[instruction(...)]` on the accounts struct to access instruction arguments during validation:

```rust
#[derive(Accounts)]
#[instruction(name: String<32>, symbol: String<10>)]
pub struct CreateNft<'info> {
    #[account(init, payer = authority, metadata::name = name, metadata::symbol = symbol)]
    pub metadata: &'info mut Account<Metadata>,
    pub authority: &'info mut Signer,
}
```

## Epilogue

After the handler returns `Ok(())`, `close` operations execute for accounts marked with `close = destination`. If the handler returns an error, the epilogue is skipped.
