`#[derive(Accounts)]` generates all parsing and validation code at compile time. Account types are zero-copy -- no heap allocation or data copying.

## Account Types

### `Account<T>`

Program-owned data account. Validates owner and discriminator. Provides `Deref`/`DerefMut` to the zero-copy type for field access.

```rust
#[derive(Accounts)]
pub struct Deposit<'info> {
    #[account(mut, has_one = authority)]
    pub vault: &'info mut Account<Vault>,
    pub authority: &'info Signer,
}
```

### `Signer`

Must have the `is_signer` flag set. No other checks.

```rust
pub maker: &'info mut Signer,
```

### `UncheckedAccount`

No validation. Use for accounts passed through to CPIs or validated manually.

```rust
#[account(mut, seeds = [b"vault", user], bump)]
pub vault: &'info mut UncheckedAccount,
```

### `SystemAccount`

Owned by the System program. Used for SOL-holding accounts with no program data.

```rust
pub destination: &'info SystemAccount,
```

### `Program<T>`

Executable program account. Validates the executable flag and address against `T::ID`.

```rust
pub token_program: &'info Program<Token>,
pub system_program: &'info Program<System>,
```

### `Interface<T>`

Like `Program<T>`, but validates against multiple allowed addresses. Use when an instruction supports both Token and Token-2022.

```rust
pub token_program: &'info Interface<TokenProgramInterface>,
```

### `Sysvar<T>`

Validates the sysvar address. Provides `Deref` to the inner type for field access.

```rust
pub rent: &'info Sysvar<Rent>,
```

### Optional Accounts

Wrap any account type in `Option`. If the account at that position has an all-zero address, it is treated as `None`.

```rust
pub optional_authority: Option<&'info Signer>,
```

## Mutability

- `&'info Account<T>` -- read-only access
- `&'info mut Account<T>` -- mutable access, account must be writable

If an account is declared `&mut` but not passed as writable in the transaction, parsing fails with `ProgramError::Immutable`.

## The `#[account]` Macro

The `#[account]` macro transforms a struct into a zero-copy account type, generating a companion struct, discriminator validation, owner checks, and typed accessors.

### Static Accounts

For accounts with only fixed-size fields:

```rust
#[account(discriminator = 1)]
pub struct Escrow {
    pub maker: Address,
    pub mint_a: Address,
    pub mint_b: Address,
    pub maker_ta_b: Address,
    pub receive: u64,
    pub bump: u8,
}
```

This generates:
- A companion struct `EscrowZc` with Pod-typed fields (alignment 1)
- `Discriminator`, `Owner`, and `Space` impls
- Discriminator validation
- `set_inner()` method for initializing all fields at once

### Dynamic Accounts

Accounts can contain variable-length fields using `String` and `Vec` marker types:

```rust
#[account(discriminator = 1)]
pub struct MultisigConfig<'a> {
    pub creator: Address,
    pub threshold: u8,
    pub bump: u8,
    pub label: String<'a, 32>,       // u32 prefix (default), max 32 bytes
    pub signers: Vec<'a, Address, 10>, // u32 prefix (default), max 10 elements
}
```

`String<P, MAX>` and `Vec<T, P, MAX>` are marker types recognized by `#[account]`. `P` is the length prefix type (`u8`, `u16`, or `u32`, defaulting to `u32`). The `'a` lifetime form (`String<'a, MAX>`) is backward-compatible shorthand that defaults to a `u32` prefix.

Each dynamic field generates:
- A getter (e.g., `label() -> &str`, `signers() -> &[Address]`)
- A raw accessor (`label_raw() -> RawEncoded`) for CPI pass-through without re-encoding
- A setter (`set_label(&mut self, payer, value)`) that handles reallocation
- For Vec fields, a mutable accessor (`signers_mut() -> &mut [Address]`)

### Discriminator Rules

- **All-zero discriminators are rejected at compile time** -- prevents uninitialized accounts from passing validation.
- **Discriminators starting with 0xFF are rejected** -- reserved for the event protocol.
- Discriminator values can be any length, but all accounts within a program should use consistent lengths.

## `#[derive(Accounts)]` Attributes

Field-level `#[account(...)]` attributes control validation.

### Core Attributes

| Attribute | Description |
|-----------|-------------|
| `mut` | Account must be writable. Implied by `init`, `close`, and `realloc`. |
| `signer` | Account must be a transaction signer. Automatically inferred for `Signer` types. |
| `dup` | Allow this account to be a duplicate of a previous account in the list. By default, Quasar requires all accounts to be distinct (non-borrowed). |

### Initialization

| Attribute | Description |
|-----------|-------------|
| `init` | Create the account via CPI to the System program. Requires `payer` and either `space` or auto-computed space. |
| `init_if_needed` | Create the account only if it does not already exist. The account must be writable and have a payer. |
| `payer = <field>` | The account that pays for rent when initializing. Must reference another field in the struct. |
| `space = <expr>` | Override the auto-computed space for `init`. |

Example:

```rust
#[account(init, payer = maker, seeds = [b"escrow", maker], bump)]
pub escrow: &'info mut Account<Escrow>,
```

### PDA Seeds and Bumps

| Attribute | Description |
|-----------|-------------|
| `seeds = [expr, ...]` | PDA seed expressions. Account field references are automatically converted to their address bytes. |
| `bump` | Find the canonical bump seed. The discovered bump is stored in the `Bumps` struct. |
| `bump = <expr>` | Use a known bump value (e.g., `bump = escrow.bump`). Verifies the PDA address instead of searching. |

When `seeds` and `bump` are used together, the macro generates PDA verification code. See [Program Derived Addresses](./pda) for details.

### Ownership and Address Checks

| Attribute | Description |
|-----------|-------------|
| `address = <expr>` | Validate that the account's address matches the given expression. |
| `has_one = <field>` | Validate that a field stored in the account data matches the address of another account in the struct. |
| `constraint = <expr>` | Arbitrary boolean constraint evaluated after account construction. |

### Custom Error Syntax

`has_one`, `constraint`, and `address` support a custom error variant with `@`:

```rust
#[account(
    has_one = maker @ MyError::InvalidMaker,
    constraint = escrow.receive > 0 @ MyError::ZeroReceiveAmount,
    address = EXPECTED_ADDRESS @ MyError::WrongAddress,
)]
pub escrow: &'info mut Account<Escrow>,
```

Without the `@` annotation, these checks return the framework's default errors (`HasOneMismatch`, `ConstraintViolation`, `AddressMismatch`).

### Closing Accounts

| Attribute | Description |
|-----------|-------------|
| `close = <field>` | Close this account after the handler returns. Zeros the discriminator, drains lamports to the destination, and reassigns to the System program. |

Close runs in the epilogue, so you can still read the account's data during execution.

```rust
#[account(close = taker, seeds = [b"escrow", maker], bump = escrow.bump)]
pub escrow: &'info mut Account<Escrow>,
```

### Reallocation

| Attribute | Description |
|-----------|-------------|
| `realloc = <expr>` | Resize the account to the given number of bytes. Adjusts lamports for rent exemption. |
| `realloc::payer = <field>` | The account that pays for (or receives refunds from) the resize. |

### Token and Mint Attributes

The `quasar-spl` crate provides specialized attributes for token accounts:

| Attribute | Description |
|-----------|-------------|
| `token::mint = <field>` | Validate (or initialize) the token account's mint. |
| `token::authority = <field>` | Validate (or initialize) the token account's authority. |
| `associated_token::mint = <field>` | For associated token account derivation. |
| `associated_token::authority = <field>` | For associated token account derivation. |
| `associated_token::token_program = <field>` | Specify which token program to use. |
| `mint::decimals = <expr>` | Set mint decimals during initialization. |
| `mint::authority = <field>` | Set mint authority during initialization. |
| `mint::freeze_authority = <field>` | Set freeze authority during initialization. |

## The Bumps Struct

For a struct named `Take` with PDA fields, the macro generates `TakeBumps` with one `u8` field per PDA and `*_seeds()` methods for CPI signing:

```rust
// Generated by #[derive(Accounts)] for Take
pub struct TakeBumps {
    pub escrow: u8,
}

impl TakeBumps {
    pub fn escrow_seeds(&self) -> [Seed; 3] {
        // Returns [b"escrow", maker_address, &[bump]]
    }
}
```

Usage:

```rust
pub fn withdraw_tokens_and_close(&mut self, bumps: &TakeBumps) -> Result<(), ProgramError> {
    let seeds = bumps.escrow_seeds();

    self.token_program
        .transfer(self.vault_ta_a, self.taker_ta_a, self.escrow, self.vault_ta_a.amount())
        .invoke_signed(&seeds)?;

    self.vault_ta_a
        .close(self.token_program, self.taker, self.escrow)
        .invoke_signed(&seeds)
}
```

## Account Parsing Pipeline

When an instruction is invoked:

1. **Header validation**: Each account's flags (signer, writable, executable, duplicate) are checked against expectations. Missing signatures, non-writable accounts, and unexpected duplicates fail here.

2. **Typed construction**: Each field is constructed from its `AccountView`.

3. **PDA verification and initialization**: `init` / `init_if_needed` accounts are created via CPI. PDA addresses are verified for `seeds` + `bump` fields.

4. **Constraint evaluation**: `has_one`, `constraint`, and `address` checks run in declaration order.

5. **Epilogue**: After the handler returns, `close` operations execute.
