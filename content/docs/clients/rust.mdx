title: Rust Client
description: Using the auto-generated Rust client for off-chain instruction building and testing.

Quasar generates Rust client code in two forms: an inline `client` module inside your program crate (non-SBF targets only), and a standalone crate at `target/client/rust/<name>-client/`. Both produce typed instruction builders that return `solana_instruction::Instruction` values, ready for `solana-client`, Mollusk, or any Solana test harness.

## The inline client module

The `#[program]` macro generates a `client` submodule gated behind `#[cfg(not(any(target_arch = "bpf", target_os = "solana")))]`, so it only compiles for off-chain targets (tests, CLI tools, scripts).

For each instruction, the module contains a struct that implements `From<T> for Instruction`. Here is the expanded output for the vault program:

```rust
// Generated by #[program] + #[derive(Accounts)]
pub mod client {
    use super::*;

    pub struct DepositInstruction {
        pub user: Address,
        pub vault: Address,
        pub system_program: Address,
        pub amount: u64,
    }

    impl From<DepositInstruction> for quasar_lang::client::Instruction {
        fn from(ix: DepositInstruction) -> quasar_lang::client::Instruction {
            let accounts = vec![
                quasar_lang::client::AccountMeta::new(ix.user, true),       // mut Signer
                quasar_lang::client::AccountMeta::new(ix.vault, false),     // mut
                quasar_lang::client::AccountMeta::new_readonly(ix.system_program, false),
            ];
            let data = quasar_lang::client::build_instruction_data(
                &[0], // discriminator
                |_data| {
                    quasar_lang::client::WriteBytes::write_bytes(&ix.amount, _data);
                },
            );
            quasar_lang::client::Instruction {
                program_id: crate::ID,
                accounts,
                data,
            }
        }
    }

    pub struct WithdrawInstruction {
        pub user: Address,
        pub vault: Address,
        pub amount: u64,
    }

    impl From<WithdrawInstruction> for quasar_lang::client::Instruction {
        fn from(ix: WithdrawInstruction) -> quasar_lang::client::Instruction {
            let accounts = vec![
                quasar_lang::client::AccountMeta::new(ix.user, true),
                quasar_lang::client::AccountMeta::new(ix.vault, false),
            ];
            let data = quasar_lang::client::build_instruction_data(
                &[1], // discriminator
                |_data| {
                    quasar_lang::client::WriteBytes::write_bytes(&ix.amount, _data);
                },
            );
            quasar_lang::client::Instruction {
                program_id: crate::ID,
                accounts,
                data,
            }
        }
    }
}
```

Struct names follow the pattern `<PascalCaseInstructionName>Instruction`. Each struct has a field for every account (as `Address`) plus every instruction argument. Import as `my_program::client::*`.

### Account metadata

`AccountMeta` for each field is derived from the accounts struct:

- **Writable**: `&'info mut T` or `#[account(mut)]` produces `AccountMeta::new()`.
- **Signer**: `Signer` produces `is_signer = true`.
- **Read-only, non-signer**: Everything else produces `AccountMeta::new_readonly()` with `is_signer = false`.

### Instruction data

Instruction data consists of discriminator bytes followed by arguments serialized via the `WriteBytes` trait. Supported types:

| Rust type | Encoding |
|-----------|----------|
| `u8`, `u16`, `u32`, `u64`, `u128` | Little-endian bytes |
| `i8`, `i16`, `i32`, `i64`, `i128` | Little-endian bytes |
| `bool` | Single byte (0 or 1) |
| `Address` | 32 bytes |
| `[u8; N]` | Raw N bytes |
| `Vec<T>` | u32 LE length prefix + serialized elements |
| `TailBytes` | Raw bytes appended with no length prefix |

### Remaining accounts

If your instruction uses `CtxWithRemaining<T>`, the generated struct includes an additional `remaining_accounts: Vec<AccountMeta>` field appended after the fixed accounts.

## Using the inline client in tests

Here is a deposit test from the vault example (`examples/vault/src/tests.rs`) using Mollusk:

```rust
extern crate std;

use {
    crate::idl_client::{DepositInstruction, WithdrawInstruction},
    mollusk_svm::{program::keyed_account_for_system_program, Mollusk},
    quasar_lang::prelude::*,
    solana_account::Account,
    solana_instruction::Instruction,
};

fn setup() -> Mollusk {
    Mollusk::new(&crate::ID, "../../target/deploy/quasar_vault")
}

#[test]
fn test_deposit() {
    let mollusk = setup();

    let (system_program, system_program_account) = keyed_account_for_system_program();

    let user = Address::new_unique();
    let user_account = Account::new(10_000_000_000, 0, &system_program);

    let (vault, _vault_bump) =
        Address::find_program_address(&[b"vault", user.as_ref()], &crate::ID);
    let vault_account = Account::new(0, 0, &system_program);

    let deposit_amount: u64 = 1_000_000_000;

    let instruction: Instruction = DepositInstruction {
        user,
        vault,
        system_program,
        amount: deposit_amount,
    }
    .into();

    let result = mollusk.process_instruction(
        &instruction,
        &[
            (user, user_account.clone()),
            (vault, vault_account.clone()),
            (system_program, system_program_account.clone()),
        ],
    );

    assert!(
        result.program_result.is_ok(),
        "deposit failed: {:?}",
        result.program_result
    );

    let user_after = result.resulting_accounts[0].1.lamports;
    let vault_after = result.resulting_accounts[1].1.lamports;

    assert_eq!(user_after, 10_000_000_000 - deposit_amount);
    assert_eq!(vault_after, deposit_amount);
}
```

Construct the instruction struct, convert it with `.into()`, and pass it to the test harness.

Note: the vault example imports from `crate::idl_client` (a hand-written client). For the auto-generated client, use `use crate::client::*;` or from an external crate, `use my_program::client::*;`.

## The standalone client crate

Running `quasar idl` also generates a standalone Rust crate at:

```
target/
  client/
    rust/
      <name>-client/
        Cargo.toml
        src/
          lib.rs
```

This crate contains the same instruction builders as a standalone library. Use it when you have a CLI tool or test suite that lives outside the program crate.

The generated `Cargo.toml` (versions read from your program's `Cargo.toml`):

```toml
[package]
name = "<name>-client"
version = "<version>"
edition = "2021"

[dependencies]
solana-instruction = "..."
solana-address = "..."
```

The standalone crate does not depend on your program crate.

## Integration with solana-client

Combine the generated instructions with `solana-client` to send transactions to a cluster:

```rust
use {
    solana_client::rpc_client::RpcClient,
    solana_sdk::{
        signature::{Keypair, Signer},
        transaction::Transaction,
    },
    solana_instruction::Instruction,
};

let client = RpcClient::new("http://localhost:8899");
let payer = Keypair::new();

// Build the instruction
let ix: Instruction = DepositInstruction {
    user: payer.pubkey(),
    vault: vault_pda,
    system_program: solana_sdk::system_program::id(),
    amount: 1_000_000_000,
}.into();

// Build and send the transaction
let recent_blockhash = client.get_latest_blockhash()?;
let tx = Transaction::new_signed_with_payer(
    &[ix],
    Some(&payer.pubkey()),
    &[&payer],
    recent_blockhash,
);
client.send_and_confirm_transaction(&tx)?;
```

## How code generation works

Client code is generated at two levels:

- **Compile-time**: `#[derive(Accounts)]` and `#[program]` generate the inline `client` module during compilation, so it is type-checked alongside your program.
- **CLI-time**: `quasar idl` parses your program source and writes the standalone Rust crate and TypeScript clients. The standalone crate depends only on `solana-instruction` and `solana-address`.

Both paths produce identical instruction encodings. Use the inline module for tests within the program crate and the standalone crate for external consumers.

## The WriteBytes trait

Instruction data serialization uses the `WriteBytes` trait in `quasar_lang::client`:

```rust
pub trait WriteBytes {
    fn write_bytes(&self, buf: &mut Vec<u8>);
}
```

Implemented for all integer types (little-endian), `bool`, `Address`, `[u8; N]`, `Vec<T>` (u32 length prefix + elements), and `TailBytes` (raw bytes, no length prefix).

## Custom client code

You can also write a manual client module. The vault example does this with `src/idl_client.rs`:

```rust
// src/idl_client.rs
use {
    alloc::vec,
    quasar_lang::prelude::*,
    solana_instruction::{AccountMeta, Instruction},
};

pub struct DepositInstruction {
    pub user: Address,
    pub vault: Address,
    pub system_program: Address,
    pub amount: u64,
}

impl From<DepositInstruction> for Instruction {
    fn from(ix: DepositInstruction) -> Instruction {
        let accounts = vec![
            AccountMeta::new(ix.user, true),
            AccountMeta::new(ix.vault, false),
            AccountMeta::new_readonly(ix.system_program, false),
        ];
        let mut data = vec![0];
        data.extend_from_slice(&ix.amount.to_le_bytes());
        Instruction {
            program_id: crate::ID,
            accounts,
            data,
        }
    }
}
```

The vault includes it conditionally:

```rust
#[cfg(test)]
mod idl_client;
#[cfg(test)]
mod tests;
```

This is useful during development before the IDL pipeline is integrated, or when you need custom logic (PDA derivation, batched instructions) that the generator does not handle.
